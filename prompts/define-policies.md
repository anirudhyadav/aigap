# define-policies.md

**Use when:** You have a policy document (PDF, Word, Markdown, Confluence export, or prose) and
need to bootstrap `.aigap/POLICIES.md` — the living guardrail spec for your AI system.

**How to use:**
1. Copy everything below the `---` divider
2. Replace every `[PLACEHOLDER]` with the real content
3. Paste into any LLM (Claude, GPT-4, Gemini)
4. Commit the output as `.aigap/POLICIES.md`
5. Commit the updated `.aigap/registry.json`

**VS Code equivalent:** `aigap: Initialize from Policy Doc` (`aigap.init`)

### Instructions

**Step 1 — Extract categories**

Read the policy document and identify distinct policy groupings (e.g. Data Privacy, Safety,
Usage Constraints, Operational). Each becomes a `GC-NNN` Guardrail Category.

For each category:
- Assign the next `GC` ID from the registry (increment counter by 1)
- Write a one-line description

**Step 2 — Extract policies**

For each category, extract every individual rule, requirement, or constraint. Each becomes a
`GP-NNN` Guardrail Policy.

For each policy:
- Assign the next `GP` ID from the registry (increment by 1)
- Write: name (≤ 8 words), description (1–2 sentences, precise and enforceable)
- Assign severity: `critical` (data breach / harm risk) | `high` (compliance violation) |
  `medium` (best practice) | `low` (advisory)
- Assign an enforcement vector (`EV-NNN`) — see Step 3

**Step 3 — Define enforcement vectors**

For each unique enforcement point, create an `EV-NNN` Enforcement Vector:
- `pre-call hook` — runs before the LLM call (prompt inspection)
- `output filter` — runs after the LLM response
- `middleware` — runs at the API layer
- `test assertion` — verified in the test suite only
- `manual review` — human-in-the-loop, not automated

**Step 4 — Write the output files**

Produce both files exactly as specified in the Output Format section below.

**Status values:**
- `🔲 gap` — policy defined but no enforcement stub exists yet
- `✅ enforced` — enforcement stub committed and passing
- `⚠️ partial` — stub exists but test coverage is incomplete
- `[DEPRECATED]` — policy removed from scope (ID kept forever)

### File 2: `.aigap/registry.json`

```json
{
  "GP": [LAST_GP_NUMBER_ASSIGNED],
  "GC": [LAST_GC_NUMBER_ASSIGNED],
  "EV": [LAST_EV_NUMBER_ASSIGNED],
  "FR": 0,
  "lastUpdated": "[DATE]"
}
```

---

## ID assignment rules

1. IDs are sequential integers with zero-padding to 3 digits: `GP-001`, `GP-012`, `GP-123`
2. The registry counter holds the **last assigned** number
3. To assign the next ID: read counter → add 1 → that is the new ID → write the entry → update counter
4. Never skip numbers
5. Never reuse a number — even if a policy is deleted (mark it `[DEPRECATED]` instead)
6. Never renumber existing IDs — even during a restructure
